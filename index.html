It's simple, really simple.

Tucan has be created to make site development simple and super easy within teams (especially agile). It runs on a flat-file database to ensure both portability and speed but if you prefer SQL, you can easily write that in. The backend of Tucan is dead simple, you have pages which consist of components, which contain a single HTML (Twig template) file, variables and resources (JS, CSS). Within the backend is also Components and Variables (as mentioned) as well as Users to allow you to manage roles and give your team access.

Tucan works around you.

When I decided to build Tucan I had a number of years of frontend (and backend) knowledge behind me and had been working with a number of content management systems that really didn't work for me. Whether it was because they were bloated with lots of extra functionality I didn't need or ever use or because I had to rewrite all of my code to fit the system, often having to learn the codebase.

Getting Started

To get started with Tucan you need to download the code from the Github repository (see left) and set the *.db files located in /app/db/ to readable, you can do this in your SSH/Terminal window using chmod or through your FTP client. Next you will need to go ahead and edit your tucan.json file within /app/config/ to reflect the sites setup and extra values - from here you can turn on debug to show output errors, set the backend login URL (default is /admin), add Google Analytics tracking and a bunch of other important stuff.

When you first login, your default credentials are as follows, please change them as soon as you're in to stop anyone having admin access to your Tucan installation!

  Username: admin
  Password: admin

Once you are in you can now setup users and start setting up your website!

It's all about Components, baby.

Components are the building blocks of your site, they are slices of code that can be used through multiple pages with unique content, editable on a page by page basis, best of all it stops you repeating parts of the site and having code duplication! A dead simple example of a component is a header that might be used on your homepage, gallery and contact us page:

<!-- heroHeader -->
<div class="heroHeader">
    <img src="" title="" alt=""/>
    <h1></h1>
    <p></p>
</div>
<!-- heroHeader -->

It's always good practice to add comments into your components both above and below containing the name of your componentsso that you can easily debug from the frontend. Adding the name into your class, this makes it a lot easier to write CSS.

So, Pages.

No longer do you have to use articles, posts, containers or publications, all of your pages are, well, pages.
